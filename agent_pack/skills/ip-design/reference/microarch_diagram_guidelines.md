## 1. 配色规范

### 1.1 主色调原则

**最小化色彩使用**：
- ✅ 主要使用：黑、白、灰三色
- ✅ 框线颜色：深灰或黑色 (`#000000`, `#333333`, `#666666`)
- ✅ 填充颜色：白色或浅灰 (`#FFFFFF`, `#F5F5F5`, `#E8E8E8`)
- ⚠️ 彩色仅用于强调关键区域或区分时钟域/电源域

### 1.2 时钟域/电源域区分

当需要区分不同时钟域或电源域时：
- 使用**浅色背景**填充整个域的范围
- 推荐配色方案：
  - 时钟域 A：浅蓝 `#DAE8FC` (或 `#E1F5FE`)
  - 时钟域 B：浅绿 `#D5E8D4` (或 `#E8F5E9`)
  - 时钟域 C：浅黄 `#FFF4E6` (或 `#FFF9C4`)
  - 异步 CDC 区域：浅橙 `#FFE6CC`
- 边框使用虚线或点线，颜色与背景同色系但更深
- 域标签字体略小，放置在域边界的左上角

### 1.3 强调元素

仅在以下情况使用彩色：
- 🔴 关键路径或高风险模块：红色或橙色边框 (`#D97757`, `#FF6B6B`)
- 🟢 新增或修改的模块：绿色边框 (`#788C5D`, `#51CF66`)
- 🟡 待定或未实现模块：黄色填充 (`#FFF9C4`)

**原则：色彩必须有明确的语义，不要为了美观而滥用颜色。**

## 2. 模块与框的对应关系

### 2.1 核心规则：一个模块一个框

```
✅ 正确：每个 .v/.sv 文件对应一个独立的矩形框
- async_fifo.sv        → 一个框
- sync_bitmap.sv       → 一个框
- fifo_controller.sv   → 一个框

❌ 错误：
- 把多个模块画在一个框里
- 用一个框表示"控制逻辑"这种抽象概念
```

### 2.2 子模块的表示

当一个模块内部实例化了其他子模块时：

```
父模块框（标准边框宽度：2pt）
┌───────────────────────────────────┐
│ parent_module                     │
│                                   │
│  ┌─────────────┐  ┌─────────────┐│
│  │ sub_mod_a   │  │ sub_mod_b   ││ ← 子模块框（细边框：1pt）
│  │             │  │             ││
│  └─────────────┘  └─────────────┘│
│                                   │
└───────────────────────────────────┘
```

**边框粗细规则：**
- 顶层模块或主要模块：**2pt - 3pt**
- 子模块（被实例化）：**1pt - 1.5pt**
- 时钟域/电源域边界：**虚线 1pt**

### 2.3 实体电路原则

**每个框必须对应实际的硬件实体**，禁止画出不存在的"概念框"。

```
✅ 允许的框：
- RTL 模块 (module xxx)
- 寄存器组
- 存储器（RAM/ROM/FIFO）
- 标准单元（需明确标注，如 "Sync FF"）

❌ 禁止的框：
- "数据通路"（太抽象，应具体到是哪个模块）
- "控制逻辑"（应指明具体的 FSM 或控制器模块）
- "接口层"（应对应具体的接口模块名称）
```

### 2.4 嵌套关系的严格性

框与框的包含关系必须严格对应电路的层次结构：

```verilog
// RTL 代码
module top_module;
    sub_module_a u_a (...);
    sub_module_b u_b (...);
endmodule

// 对应的图示（必须严格匹配）
┌─ top_module ──────────┐
│  ┌─ sub_module_a ─┐   │
│  │                │   │
│  └────────────────┘   │
│  ┌─ sub_module_b ─┐   │
│  │                │   │
│  └────────────────┘   │
└───────────────────────┘
```

**严禁：**
- A 模块和 B 模块在 RTL 中是平级的，但在图中却画成嵌套关系
- C 模块实例化了 D 模块，但在图中没有包含关系

## 3. 连线规范

### 3.1 横平竖直原则

所有连接线**必须**遵循正交规则：
- ✅ 只允许水平线和垂直线
- ✅ 转角必须是 90° 直角
- ❌ 禁止斜线
- ❌ 禁止曲线（除非表示特殊的物理连接，如 PCB 走线）

### 3.2 连线路由建议

**路由优先级：**
1. **最短路径优先**：减少不必要的绕行
2. **避免重叠**：多条信号线平行走线时留出间距
3. **主干清晰**：总线或主要数据通路用加粗线条（2pt）

**推荐布局：**
```
输入端口在左侧  →  模块处理  →  输出端口在右侧
```

**信号流向：**
- 数据流：实线箭头 `→` 或 `⇒`（加粗）
- 控制信号：细实线箭头 `→`
- 时钟信号：特殊箭头或三角波形
- 复位信号：虚线箭头 `⤏`

### 3.3 总线表示

对于多 bit 信号：
- 在信号线旁标注位宽，如 `[31:0]`
- 使用加粗线条（2pt）或双线表示总线
- 如果总线拆分，在分支点标注各分支的位宽

```
     ┌─────────┐
     │         │
[31:0] ━━━━━━━━┫ Module │
     │         │
     └─────────┘
```

## 4. 框体样式规范

### 4.1 圆角设置

**统一的圆角风格**：
- 圆角半径：**3px - 6px**（稍微不尖锐即可）
- ❌ 不要使用过大的圆角（> 10px）
- ❌ 不要混用尖角和圆角

**draw.io 设置：**
```
Format → Property → Rounded = 3-6
```

### 4.2 框体尺寸

**最小尺寸要求：**
- 宽度：≥ 80px
- 高度：≥ 40px
- 如果框内有子模块，宽高应足够容纳 + 留出边距（≥ 10px）

**对齐：**
- 所有框的边缘尽量对齐（使用网格吸附）
- 同级模块的框高度尽量一致

### 4.3 文字标注

**模块名称：**
- 位置：框的顶部或左上角
- 字体：Sans-serif（推荐 Arial、Helvetica、Noto Sans）
- 字号：12pt - 14pt（顶层）、10pt - 12pt（子模块）
- 颜色：黑色 `#000000` 或深灰 `#333333`
- 加粗：顶层模块加粗，子模块正常

**端口标注：**
- 输入端口在左侧，输出端口在右侧
- 字号：8pt - 10pt
- 格式：`port_name[width]`，例如 `wr_data[31:0]`

## 5. 图例和文字说明

### 5.1 图例的必要性

每张微架构图**必须包含图例（Legend）**，说明：
- 不同线型的含义（实线、虚线、加粗线）
- 不同颜色的含义（如果使用了彩色）
- 特殊符号的含义（时钟、复位、CDC 标记等）

### 5.2 图例的位置和样式

**位置：**
- 图的右下角或左下角
- 独立的矩形框，背景白色，边框浅灰

**内容示例：**
```
┌─ 图例 ─────────────────┐
│ ━━━━  数据总线         │
│ ────  控制信号         │
│ ⤏⤏⤏⤏  时钟信号         │
│ ┄┄┄┄  复位信号         │
│ [蓝色背景] WCLK 域     │
│ [绿色背景] RCLK 域     │
└────────────────────────┘
```

### 5.3 文字描述的最小化

**原则：图表应自解释，尽量减少文字描述。**

✅ **允许的文字：**
- 模块名称
- 端口名称和位宽
- 时钟域/电源域标签
- 关键信号的流向标注

❌ **避免的文字：**
- 冗长的功能描述（应该放在规格文档中，不在图上）
- 重复的信息（如果模块名已经说明功能，不要再写一遍）
- 实现细节（如"这里使用 2 级同步"，除非必须强调）

**推荐做法：**
- 如果需要详细说明，在图的下方或另起一页写"图注"
- 图本身只保留必要的标签

## 6. 特殊元素表示

### 6.1 时钟域交叉（CDC）

**标准表示方法：**
1. 用虚线矩形框出 CDC 边界
2. 在边界上标注 "CDC" 或 "Async"
3. 跨域的信号线用虚线表示
4. 同步器（Synchronizer）必须画出为独立的框

```
┌─ WCLK Domain ──────────┐        ┌─ RCLK Domain ───────────┐
│                        │        │                         │
│  ┌─────────┐           │  CDC   │  ┌─────────┐            │
│  │ Module A │ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ▶│ Sync FF  │──▶ Module B│
│  └─────────┘           │        │  └─────────┘            │
│                        │        │                         │
└────────────────────────┘        └─────────────────────────┘
```

### 6.2 存储器（Memory）

**SRAM/RAM/FIFO 的表示：**
- 用矩形框，但填充颜色略深（浅紫或浅灰）
- 标注容量和位宽，如 `RAM 1KB (32b×256)`
- 读写端口分别标注

### 6.3 FSM（有限状态机）

如果需要在架构图中表示 FSM：
- 用圆角矩形框
- 在框内标注 "FSM: <状态机名称>"
- 状态转移图应单独绘制，不要和架构图混在一起

## 7. 工具特定设置

### 7.1 draw.io 推荐配置

**页面设置：**
- 网格大小：10px
- 启用网格吸附（Snap to Grid）
- 背景：白色

**默认样式：**
```
矩形框：
  - Fill: #FFFFFF
  - Stroke: #333333
  - Stroke Width: 2pt
  - Rounded: 5px

连接线：
  - Stroke: #333333
  - Stroke Width: 1pt
  - Line Style: Orthogonal
  - Arrow: Classic
```

**推荐库：**
- General（基本形状）
- Entity Relation（如果需要表示数据流）
- UML（如果有接口/类的概念）

### 7.2 VS Code Draw.io 扩展导出

当 `hediet.vscode-drawio` 扩展已安装时（可通过搜索已安装扩展确认），有两种导出方式：

**方式 A — `.drawio.png` 格式（推荐）**:
- 将图表文件命名为 `*_arch.drawio.png`（而非 `*_arch.drawio`）
- 该格式既是合法 PNG 图片，又内嵌了 drawio XML 元数据
- 可直接在图片查看器中预览，也可在 drawio 编辑器中继续编辑
- 对应 VS Code 命令：`hediet.vscode-drawio.convert`（将 `.drawio` 转为 `.drawio.png`）

**方式 B — Export 命令**:
- 在 VS Code 中打开 `.drawio` 文件
- 执行命令 `hediet.vscode-drawio.export`（"Export To..."）
- 选择 PNG 或 SVG 格式导出

**Agent 操作流程**:
1. 创建 `docs/*_arch.drawio`（drawio XML 格式）
2. 检查 `hediet.vscode-drawio` 是否可用（搜索已安装扩展）
3. 若可用：使用 `hediet.vscode-drawio.convert` 命令转为 `.drawio.png`
4. 若不可用：**直接报告**此问题（标注 PNG 缺失原因），不要静默跳过

⚠️ **不要降级处理**——PNG 是必需交付物。如果无法导出，报告给用户，让用户手动导出或安装扩展。

### 7.3 其他工具

**Visio：**
- 使用"流程图"模板
- 启用"自动对齐"
- 使用连接器工具确保正交路由

**Inkscape/Illustrator：**
- 使用网格和参考线
- 统一设置对象样式
- 导出为 SVG 或高分辨率 PNG

## 8. 审查清单

在完成微架构图后，使用以下清单进行自查：

- [ ] **配色**：是否主要使用黑白灰？彩色是否有明确语义？
- [ ] **模块对应**：每个框是否对应一个实际的 .v/.sv 模块？
- [ ] **层次关系**：框的嵌套是否严格对应 RTL 代码的实例化关系？
- [ ] **连线**：是否所有连线都是横平竖直的？是否有斜线或曲线？
- [ ] **圆角**：所有框的圆角是否统一？是否过大或不一致？
- [ ] **图例**：是否包含图例？图例是否清晰？
- [ ] **文字**：是否避免了冗余的文字描述？
- [ ] **可读性**：图表是否一目了然？是否需要过多解释才能理解？
- [ ] **一致性**：同一项目中的多张图是否风格统一？

## 9. 常见错误示例

### ❌ 错误 1：颜色过多

```
各种彩色框：红、橙、黄、绿、蓝、紫...
👉 问题：难以分辨哪些颜色有意义，哪些只是装饰
```

### ❌ 错误 2：概念框

```
┌─ "数据处理层" ─────────┐
│  [这不是一个具体的模块]  │
└─────────────────────────┘
👉 问题：无法对应到 RTL 代码
```

### ❌ 错误 3：斜线连接

```
  ┌────┐   ╱
  │ A  │  ╱
  └────┘ ╱
        ╱  ┌────┐
       ╱───│ B  │
           └────┘
👉 问题：不符合正交原则
```

### ❌ 错误 4：过大圆角

```
  ╭───────────╮
  │   Module  │  ← 看起来像药丸，不像电路模块
  ╰───────────╯
👉 问题：圆角过大（>15px），失去工程感
```

## 10. 总结

**核心思想：清晰 > 美观**

微架构图的目的是准确传达设计结构，而不是艺术作品。遵循以下原则：
1. **最小化色彩**：黑白灰为主
2. **严格对应**：框 = 模块，嵌套 = 实例化
3. **正交连线**：横平竖直
4. **统一风格**：圆角、线宽、字体保持一致
5. **精简文字**：图表自解释

**记住：优秀的微架构图应该让审阅者在 10 秒内理解整体结构，30 秒内理解细节。**
