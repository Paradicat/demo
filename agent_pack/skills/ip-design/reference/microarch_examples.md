```
│  ╚════════════════╝    ╲───────╱    ╚════════════════╝        │
│         ╲                  │                ╱                   │
│          ╲     [控制逻辑 - 概念框]         ╱                    │
│           ╲           (抽象)             ╱                      │
│            ╲                           ╱                        │

│              ▼─────────────────────▼                            │

```

**问题：**
- ❌ 过多文字描述
- ❌ 使用了圆形、斜线等非正交元素
- ❌ 颜色过多且无明确语义
- ❌ "控制逻辑"是抽象概念，不对应具体模块
- ❌ 圆角过大
- ❌ 没有明确的时钟域区分

## 示例 2：流水线处理器微架构

### ✅ 正确示例

```
┌─ 处理器核心 (Processor Core) ────────────────────────────────────────────────┐
│                                                                               │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐│
│  │   取指    │───▶│   译码    │───▶│   执行    │───▶│   访存    │───▶│  写回   ││
│  │   (IF)   │    │   (ID)   │    │   (EX)   │    │   (MEM)  │    │  (WB)  ││
│  │          │    │          │    │          │    │          │    │        ││
│  │ if_stage │    │ id_stage │    │ ex_stage │    │ mem_stage│    │wb_stage││
│  └────┬─────┘    └────┬─────┘    └────┬─────┘    └────┬─────┘    └────┬────┘│
│       │               │               │               │               │     │
│       │ pc[31:0]      │ inst[31:0]    │ alu_out      │ mem_data     │     │
│       │               │               │               │               │     │
│  ┌────▼─────────────────────────────────────────────────────────────▼─────┐ │
│  │                         寄存器堆 (regfile)                             │ │
│  │                     32 × 32-bit 通用寄存器                             │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘

┌─ 图例 ───────────────┐
│  ━━━━━━  数据通路    │
│  ──────  控制信号    │
└──────────────────────┘
```

**要点：**
- ✅ 每个流水线阶段对应一个具体的模块文件

- ✅ 共享的寄存器堆单独画出
- ✅ 连线简洁，没有过多交叉

## 反例总结

### 常见错误汇总

| 错误类型 | 描述 | 正确做法 |
|---------|------|---------|
| **颜色滥用** | 使用 5 种以上颜色，无明确语义 | 黑白灰为主，彩色只用于时钟域或强调 |
| **概念框** | 画出"逻辑层""接口层"等抽象框 | 每个框对应具体的 .v/.sv 文件 |
| **斜线连接** | 用斜线连接模块 | 只用横线和竖线，90° 转角 |
| **圆角过大** | 圆角半径 > 15px | 圆角半径 3-6px |
| **文字过多** | 大段文字描述功能 | 只标注必要的模块名和信号名 |
| **层次混乱** | 框的嵌套关系与 RTL 不符 | 严格按照 module 实例化关系嵌套 |
| **无图例** | 图中没有图例说明 | 每张图必须有图例 |
| **信号未标注** | 总线没有位宽，信号没有名称 | 所有总线标注位宽，关键信号标注名称 |

## 进阶技巧

### 1. 大型设计的分层展示

对于复杂设计，可以分为多层图：
- **L0 顶层架构图**：只显示主要模块和接口
- **L1 子系统图**：展开某个子系统的内部结构
- **L2 模块细节图**：单个模块的内部实现（如果必要）

### 2. 使用参考坐标

在复杂图中，可以用字母数字坐标（如 A1, B2）标注区域，方便讨论。

### 3. 版本控制

在图的右下角标注：
- 版本号（v1.0, v2.0）
- 日期（2026-02-17）
- 作者

## 结论

通过对比正确示例和错误示例，可以看出：
- **简洁的配色**让图表更专业
- **严格的模块对应**让图表可追溯
- **正交的连线**让图表更清晰
- **适当的文字**让图表易读但不冗余

**记住：优秀的微架构图应该让读者一眼看出设计结构，而不是花时间解读图的表达方式。**
